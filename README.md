# AI-Driven AIOps Agent Prototype

## Demo

See `Demo.mp4` in this repository for a short walkthrough of the project and a live demonstration of the agent running.

## Overview

This project is a prototype AIOps agent that simulates how AI can assist with IT operations. The system continuously monitors telemetry from a simulated microservice environment, detects anomalies using a machine learning model, performs simple root cause analysis using metrics, logs, and traces, and then decides whether to automatically remediate the issue or escalate it to a human.

The goal of this project is not to build a production system, but to demonstrate how AI techniques can be applied to operational monitoring and incident response.

---

## Architecture

The system is composed of several components that together simulate an autonomous operations loop.

- **main.py**  
  Entry point that initializes the environment, agent, and orchestrator, and runs the simulation loop.

- **orchestrator.py**  
  Coordinates the system. It collects telemetry, calls the agent for decisions, executes remediation actions, and records artifacts.

- **agent.py**  
  Contains the decision logic. The agent evaluates anomaly signals, performs lightweight root cause analysis using metrics, logs, and traces, and selects remediation actions.

- **anomaly.py**  
  Implements the anomaly detection model used to learn baseline system behavior and identify unusual patterns.

- **sim_env.py**  
  Simulates a microservice environment and injects faults such as memory pressure, connection failures, or authentication errors.

- **telemetry.py**  
  Defines the telemetry structures used to represent metrics, logs, and trace information.

- **config.yaml**  
  Configuration file for thresholds, remediation settings, and other runtime parameters.

- **artifacts/**  
  Stores snapshots of telemetry and incident timelines produced during execution.

The overall workflow is:

1. Collect telemetry from the simulated environment  
2. Detect anomalies using the anomaly detector  
3. Analyze logs and traces to generate a root cause hypothesis  
4. Execute remediation actions when appropriate  
5. Escalate to a human when automation is unsafe  
6. Record incident artifacts

---

## AI Techniques and Decision Logic

The agent uses two main ideas from AI and AIOps systems.

**Unsupervised anomaly detection**

The anomaly detector learns baseline behavior during an initial warmup phase and then flags unusual telemetry patterns. This allows the system to identify potential incidents without requiring labeled training data.

**Multi-signal reasoning**

When an anomaly is detected, the agent correlates multiple signals:

- system metrics (CPU, memory, latency, error rate)
- log signals (e.g., OOM errors, authentication failures)
- trace errors from service dependencies

These signals are combined to produce a root cause hypothesis. Based on the hypothesis, the agent chooses one of several actions:

- restart a service
- scale a service
- escalate the incident by creating a ticket

Operational guardrails such as incident suppression and ticket deduplication are included to prevent alert storms and repeated automation.

## Setup Instructions

1. Clone the repository.

2. Create and activate a virtual environment.

python3 -m venv .venv
source .venv/bin/activate


3. Install dependencies.

pip install -r requirements.txt


4. Run the agent.

python main.py



The system will run a simulated monitoring loop and print incidents and remediation actions to the console.

Artifacts describing incidents and telemetry snapshots will be written to the `artifacts/` directory.

---

## Assumptions

- The system operates on a simulated microservice environment rather than real infrastructure.
- Telemetry signals are generated by the simulator rather than external monitoring systems.
- Remediation actions such as restart or scaling are simulated through environment functions.
- The anomaly detection model is trained only on data observed during the simulation warmup phase.

## Future Improvements

Several extensions could make this prototype more realistic:

- Integration with real telemetry sources such as Prometheus or OpenTelemetry
- More advanced anomaly detection models and feature engineering
- Use of LLMs for richer root cause explanations
- Reinforcement learning for selecting remediation strategies
- Visualization dashboards for telemetry and incident timelines
- Integration with real ticketing systems or incident management platforms
